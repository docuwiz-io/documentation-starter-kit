# Documentation Templates by Docuwiz

[Docuwiz.io](https://docuwiz.io) is a documentation platform built for modern software teams. It brings together structured writing, API design, and publishing into a single workflow — so your docs stay as close to your code as your codebase does.

Whether you're documenting a new microservice, writing onboarding tutorials for developers, or keeping stakeholders in the loop with release notes, Docuwiz gives you the scaffolding to do it consistently and fast.

---

## Start from a template — in one click

Inside the Docuwiz UI, every template in this repository is available as a starting point. Pick a doc type, hit **"Use Template"**, and you're dropped into a pre-structured document ready to fill in — no blank page, no second-guessing the structure.

Templates are opinionated by design. Each one is built around a specific reader, a specific goal, and a specific moment in the product lifecycle. That's what makes them reusable without feeling generic.

---

## The templates

### Getting Started Tutorial

**File:** `Tutorial_ Getting Started _template_byDocuwiz.md`

A hands-on guide for first-time users. This template is structured around a real outcome — by the end of it, the reader has built or configured something working. It's not a feature tour. It's an onboarding experience.

Use it when you're introducing a new product, a new module, or a major workflow to someone who has never touched it before. It works especially well alongside short demo videos embedded directly in the doc.

---

### How-to Guide

**File:** `How-to Guide_ Task-Based_template_byDocuwiz.md`

Task-focused. Aimed at existing users who already know what they're doing and just need to get a specific thing done. Think of it as a recipe — clear ingredients, clear steps, clear result.

Use it when a user knows what they want to accomplish but might not know the exact sequence of actions. Good for configuration tasks, migrations, integrations, and anything that requires precision without a lot of background explanation.

---

### Conceptual Guide (Explanation)

**File:** `Explanation_ Conceptual Guide_template_byDocuwiz.md`

Answers the "why does this work this way?" question. This template helps readers build a mental model of a feature or system behaviour — not how to use it, but how to understand it.

Use it when a concept is abstract, non-obvious, or frequently misunderstood. Strong conceptual docs reduce support tickets and make your how-to guides much easier to follow because readers already understand the underlying logic.

---

### Reference: Technical Lookup

**File:** `Reference_ Technical Lookup_template_byDocuwiz.md`

Dense, precise, no storytelling. This is the doc developers open in a second tab while they're actively building. It covers field definitions, allowed values, parameter types, defaults, and validation rules.

Use it for APIs, config files, SDK options, and anything where exact values matter. Reference docs don't need to explain — they need to be complete and scannable.

---

### Architecture Overview

**File:** `Architecture Overview_template_byDocuwiz.md`

A high-level map of how a system is put together — components, responsibilities, data flow, and security boundaries. Not a setup guide, not a marketing doc. Just a clear structural picture that helps engineers orient themselves quickly.

Use it when onboarding new team members, kicking off an integration, or documenting system design decisions for future reference. It's also useful before a major refactor, as a baseline to reason from.

---

### Best Practices & Guidelines

**File:** `Best Practices & Guidelines_template_byDocuwiz.md`

Opinionated but not mandatory. This template captures the collective wisdom of your team — naming conventions, configuration patterns, security hygiene, and common things to avoid.

Use it when you want to raise the floor for how people use your product or internal tools. The best version of this doc reads like advice from a senior engineer who's already made the mistakes you're trying to prevent.

---

### Changelog

**File:** `Changelog_template_byDocuwiz.md`

A versioned, chronological log of what changed, when, and in which release. Organized into Added, Changed, and Fixed sections so readers can scan for what matters to them.

Use it for internal engineering communication and developer-facing products. Changelogs build trust — they show that the product is actively maintained and that breaking changes are communicated clearly.

---

### Release Notes

**File:** `Release Notes_template_byDocuwiz.md`

Customer-facing and broader in tone than a changelog. Release notes explain what changed and why it matters — not just to engineers, but to anyone using or evaluating the product.

Use it when announcing a new version to customers, stakeholders, or a public audience. This template covers new features, enhancements, bug fixes, and known issues, giving readers a complete picture of a release.

---

### Troubleshooting & FAQs

**File:** `Troubleshooting & FAQs_template_byDocuwiz.md`

A structured problem-solving guide. This template is built around the reader who's stuck — something isn't working, and they need a fast path to resolution without opening a support ticket.

Use it proactively as you ship features, especially ones with non-obvious failure modes. A well-maintained troubleshooting page significantly reduces support load and is often the most-visited page in any developer doc site.

## Built for developers — with Docuwiz

If you're a developer, Docuwiz fits into how you already work.

**Docs as code.** Write and manage documentation in Markdown, version it in Git, and review it in pull requests alongside the code it documents. No CMS logins, no "who has edit access" conversations.

**AI-powered API design.** Docuwiz includes a built-in AI enhancement layer for OpenAPI Specification (OAS) files. Paste in a bare spec and Docuwiz enriches it with descriptions, examples, parameter constraints, and enum values — turning a functional API file into production-quality documentation without the manual grind.

**One-click publish.** When your docs are ready, publish them directly from the Docuwiz UI to a hosted developer portal. No build pipeline to configure, no deployment scripts.
